PROJECT OVERVIEW
================================================================================
StudyNest is a full-stack Learning Management System (LMS) designed for 
engineering students to take semester-wise tests, track their progress, and 
earn certificates upon successful completion. The platform supports two user 
roles: Students and Tutors, with role-based access control and comprehensive 
module management capabilities.

TECH STACK
================================================================================

BACKEND:
--------
- Framework: Spring Boot 3.5.5
- Language: Java 17
- Database: H2 (In-Memory) - Production ready for MySQL/PostgreSQL
- Security: Spring Security + JWT Authentication
- ORM: Spring Data JPA (Hibernate)
- PDF Generation: Apache PDFBox 2.0.29
- Build Tool: Maven
- Architecture: RESTful API with MVC Pattern

FRONTEND:
---------
- Framework: React 18.3.1
- Build Tool: Vite 5.4.10
- Routing: React Router DOM 6.22.0
- UI Library: React Bootstrap 5.3.3
- HTTP Client: Axios 1.6.7
- Icons: Lucide React + React Icons
- Styling: Custom CSS + Bootstrap

KEY FEATURES
================================================================================

1. AUTHENTICATION & AUTHORIZATION
   - JWT-based stateless authentication

DATABASE SCHEMA
================================================================================

ENTITIES:
---------
1. Student
   - id (PK), name, email (unique), password (encrypted)
   - rollNo, prn, semester, mobile
   - createdAt, updatedAt
   - Relationships: One-to-Many with StudentResult

2. Tutor
   - id (PK), name, email (unique), password (encrypted)
   - mobile, specialization
   - createdAt, updatedAt

3. Module
   - id (PK), name, code (unique), semester
   - description, imageUrl, pdfUrl, videoLink, externalLinks
   - Relationships: One-to-Many with Question

4. Question
   - id (PK), questionText
   - option1, option2, option3, option4
   - correctAnswer (stores "option1", "option2", etc.)
   - Relationships: Many-to-One with Module

5. StudentResult
   - id (PK), score, totalQuestions, percentage
   - status (PASSED/FAILED), completedAt
   - Relationships: Many-to-One with Student, Many-to-One with Module

DATA SEEDING:
-------------
- 25 modules (5 per semester, semesters 2-6)
- 250 questions (10 per module)
- Test user: test@studynest.com / test123 (Semester 6 completed)

API ENDPOINTS
================================================================================

AUTHENTICATION (/api/auth):
---------------------------
POST /register/student - Register new student
POST /register/tutor - Register new tutor
POST /login - Login (returns JWT token)

MODULES (/api/modules):
-----------------------
GET / - Get all modules
GET /semester/{semester} - Get modules by semester
GET /{id} - Get module by ID
POST / - Create module (TUTOR only)
PUT /{id} - Update module (TUTOR only)
DELETE /{id} - Delete module (TUTOR only)

QUESTIONS (/api/questions):
---------------------------
GET /module/{moduleId} - Get questions for a module
POST / - Create question (TUTOR only)
PUT /{id} - Update question (TUTOR only)
DELETE /{id} - Delete question (TUTOR only)

TESTS (/api/tests):
-------------------
POST /submit - Submit test answers
GET /result/{moduleId} - Get test result for a module

STUDENTS (/api/students):
-------------------------
GET /profile - Get student profile
GET /results - Get all test results
GET /progress - Get semester progress

CERTIFICATE (/api/certificate):
-------------------------------
GET /check-eligibility - Check if eligible for certificate
GET /download - Download PDF certificate

SECURITY IMPLEMENTATION
================================================================================

1. JWT AUTHENTICATION
   - Secret key stored in application.properties
   - Token expiration: 24 hours (86400000 ms)
   - Token structure: Header.Payload.Signature
   - Stored in localStorage on frontend
   - Sent in Authorization header: "Bearer {token}"

2. PASSWORD SECURITY
   - BCrypt hashing algorithm
   - Salt rounds: 10 (default)
   - Passwords never stored in plain text

3. AUTHORIZATION
   - Method-level security with @PreAuthorize
   - Role-based access: ROLE_STUDENT, ROLE_TUTOR
   - Protected routes on frontend with ProtectedRoute component

4. CORS CONFIGURATION
   - Allowed origins: http://localhost:5173 (frontend)
   - Allowed methods: GET, POST, PUT, DELETE
   - Credentials allowed for cookie/session support

5. SECURITY FILTERS
   - JwtAuthenticationFilter: Validates JWT on each request
   - CustomUserDetailsService: Loads user from database
   - SecurityConfig: Configures security rules

LAYERED ARCHITECTURE:
---------------------
1. Controller Layer (@RestController)
   - Handles HTTP requests/responses
   - Input validation
   - Delegates to service layer

2. Service Layer (@Service)
   - Business logic implementation
   - Transaction management
   - Delegates to repository layer

3. Repository Layer (JpaRepository)
   - Data access abstraction
   - CRUD operations
   - Custom query methods

4. Entity Layer (@Entity)
   - JPA entities mapped to database tables
   - Relationships defined
   - Lombok for boilerplate reduction

5. DTO Layer (Data Transfer Objects)
   - Decouples entities from API responses
   - Prevents over-fetching
   - Clean API contracts

6. Security Layer
   - JWT utilities
   - Authentication filters
   - User details service

7. Exception Handling
   - Global exception handler (@RestControllerAdvice)
   - Custom exceptions
   - Consistent error responses

2. TECHNICAL DEEP-DIVE QUESTIONS
================================================================================

BACKEND QUESTIONS:
------------------

Q: Explain your authentication flow.
A: "I implemented JWT-based stateless authentication:
1. User submits credentials to /api/auth/login
2. Backend validates credentials using BCrypt
3. If valid, generates JWT token with user details and role
4. Token sent to frontend, stored in localStorage
5. Frontend includes token in Authorization header for protected requests
6. JwtAuthenticationFilter intercepts requests, validates token
7. If valid, sets SecurityContext with user details
8. @PreAuthorize annotations check role-based permissions"

Q: How did you implement role-based access control?
A: "I used Spring Security's method-level security:
1. Each user has a role (STUDENT or TUTOR) stored in database
2. JWT token includes role in claims
3. Controllers use @PreAuthorize("hasRole('TUTOR')") annotations
4. Frontend has ProtectedRoute component checking user role
5. Unauthorized access returns 403 Forbidden
Example: Module CRUD operations are restricted to TUTOR role only."

Q: Explain your database relationships.
A: "I designed 5 entities with these relationships:
1. Student → StudentResult (One-to-Many): One student has many test results
2. Module → Question (One-to-Many): One module has many questions
3. Module → StudentResult (One-to-Many): One module has many results
4. StudentResult → Student (Many-to-One): Each result belongs to one student
5. StudentResult → Module (Many-to-One): Each result is for one module

I used JPA annotations like @OneToMany, @ManyToOne with proper cascade types."

Q: How does the certificate generation work?
A: "I implemented pure code-based PDF generation using Apache PDFBox:
1. Check eligibility: All 5 semester modules completed with ≥60% each
2. Create PDDocument with A4 landscape page
3. Draw decorative elements (borders, triangles) using PDPageContentStream
4. Add text with different fonts and sizes for hierarchy
5. Include student details, certificate number, performance stats
6. Add signature placeholders
7. Return PDF as byte array with proper Content-Type headers
8. Frontend triggers download with blob URL"

Q: How did you handle data seeding?
A: "I used two approaches:
1. data.sql: SQL INSERT statements for 25 modules (executed on startup)
2. DataSeeder.java: CommandLineRunner that:
   - Checks if data exists to avoid duplicates
   - Seeds 250 questions (10 per module) with subject-specific content
   - Creates test user with completed Semester 6 for testing
   - Uses transaction management for data consistency"

Q: Explain your exception handling strategy.
A: "I implemented centralized exception handling:
1. Custom exceptions: ResourceNotFoundException, DuplicateResourceException
2. @RestControllerAdvice with @ExceptionHandler methods
3. Returns consistent ErrorResponse DTO with:
   - HTTP status code
   - Error message
   - Timestamp
4. Handles validation errors, authentication errors, and business logic errors
5. Frontend displays user-friendly error messages"

FRONTEND QUESTIONS:
-------------------

Q: How did you structure your React application?
A: "I followed a feature-based structure:
1. Components: Reusable UI components (Sidebar, TopBar, ModuleCard)
2. Pages: Feature-specific pages organized by role (student/, tutor/)
3. Services: API layer with Axios for backend communication
4. Routing: React Router with protected routes
5. State: Local state with useState, user state in App.jsx
6. Styling: Mix of Bootstrap and custom CSS for consistency"

Q: How did you implement protected routes?
A: "I created a ProtectedRoute component that:
1. Checks if user is authenticated (token exists)
2. Verifies user has required role
3. If not authenticated, redirects to /login
4. If wrong role, redirects to appropriate dashboard
5. If authorized, renders the requested component
Used in App.jsx to wrap all protected routes."

Q: How do you handle API calls?
A: "I created a service layer pattern:
1. api.js: Axios instance with base URL and interceptors
2. Request interceptor: Adds JWT token to Authorization header
3. Response interceptor: Handles 401 errors (logout user)
4. Service files (authService, moduleService, etc.): Encapsulate API calls
5. Components call service methods, not Axios directly
6. Async/await with try-catch for error handling
7. Loading states and error messages for UX"

Q: Explain your state management approach.
A: "I kept it simple without Redux:
1. User state in App.jsx (authenticated user data)
2. Passed via props to child components
3. Local state in components for UI state (loading, errors, form data)
4. Token in localStorage for persistence
5. Service layer handles API state
For a larger app, I'd consider Redux or Context API."

4. Ensured proper spacing to avoid overlaps

Q: How did you ensure code quality?
A: "I followed best practices:
1. Layered architecture for separation of concerns
2. DTOs to decouple entities from API responses
3. Lombok to reduce boilerplate code
4. Meaningful variable and method names
5. Consistent error handling
6. Transaction management for data consistency
7. Input validation at controller level
8. Logging for debugging and monitoring"

Q: How did you handle a bug?
A: "Example: Backend startup error with 'NULL not allowed for column CODE'
1. Read error logs carefully to identify root cause
2. Realized Module entity was updated with new 'code' field
3. But data.sql still had old INSERT statements
4. Updated all 25 INSERT statements with code and description
5. Tested backend startup - success
6. Documented the fix for future reference
Lesson: Always update seed data when schema changes."
